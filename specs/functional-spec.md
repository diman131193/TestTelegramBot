# Функциональная спецификация

## Команды

### `/start`

**Текущее поведение:**  
Бот сбрасывает режим консультации для чата, логирует пользователя с командой `start`, отправляет фото `files.start`, текст `texts.start` и клавиатуру выбора роли.

**Кнопки:**

- `Мастер` -> callback `master`
- `Клиент` -> callback `client`

**Acceptance criteria:**

- Пользователь получает приветственное фото и HTML-текст.
- В БД создается или обновляется запись пользователя.
- Если пользователь был в режиме консультации, режим отключается.

### `/reviews`

**Текущее поведение:**  
Бот логирует пользователя, отправляет media group из `files.reviews`, затем текст `texts.reviews` и клавиатуру с кнопкой записи.

**Acceptance criteria:**

- Если список фото отзывов пустой, бот отправляет только текст отзывов.
- Если список фото есть, бот отправляет фото перед текстом.
- После показа отзывов доступна кнопка `Запись`.

### `/load`

**Текущее поведение:**  
Служебная команда отправляет локальный файл `data/IMG_2358.jpg`, затем печатает `file_id` в stdout.

**Ограничение:**  
Файл `data/IMG_2358.jpg` отсутствует в текущем репозитории. Команда может упасть, если файл не добавлен локально.

## Callback-сценарии

### `client`

Бот сбрасывает режим консультации, логирует действие, отправляет фото `files.client`, текст `texts.client` с именем пользователя и клиентскую клавиатуру.

Клиентская клавиатура:

- `Услуги` -> `services`
- `Прайс` -> `price`
- `Отзывы` -> `reviews`
- `Бесплатная диагностика` -> `test`
- `Запись` -> внешняя ссылка `https://dikidi.ru/1723277`
- `Консультация` -> `consulting`

### `services`

Бот сбрасывает режим консультации, логирует действие, отправляет фото `files.services`, текст `texts.services` и клавиатуру услуг.

Кнопки:

- `Кератиновое восстановление` -> `keratin`
- `Ботокс для волос` -> `botox`
- `Нанопластика` -> `nanoplastic`

### `keratin`, `botox`, `nanoplastic`

Бот сбрасывает режим консультации, логирует выбранную услугу, отправляет фото и HTML-описание услуги.

После услуги показывается клавиатура:

- `Прайс` -> `price`
- `Отзывы` -> `reviews`
- `Запись` -> внешняя ссылка
- `Консультация` -> `consulting`
- `Клиент` -> `client`

### `reviews`

Бот логирует действие, отправляет отзывы через общий сценарий `get_reviews`.

### `master`

Бот сбрасывает режим консультации, логирует действие и отправляет текст `texts.master`.

### `consulting`

Бот добавляет chat ID пользователя в in-memory set `ADMIN_CHATS`, логирует действие и просит написать вопрос.

После этого следующий текст пользователя, если он не начинается с `/`, считается вопросом для консультации.

### `price`

**Текущее состояние:**  
Кнопка существует, но обработчика `callback_query(F.data == const.PRICE)` нет. Текст `price` также отсутствует в `data/texts.json`.

**Ожидаемое поведение для доработки:**

- Бот должен отвечать на `price`.
- Ответ должен содержать актуальный прайс или понятное сообщение-заглушку.
- После ответа должна быть доступна навигация: `Запись`, `Консультация`, `Клиент`.

## Консультация

### Отправка вопроса клиентом

Условие: chat ID пользователя находится в `ADMIN_CHATS`, сообщение является текстом и не начинается с `/`.

Бот отправляет администратору сообщение:

- заголовок `Новое сообщение для консультации`;
- имя и фамилию пользователя;
- username или `—`;
- ID клиентского чата;
- текст вопроса.

После отправки бот подтверждает клиенту, что вопрос передан мастеру.

### Ответ администратора

Условие: сообщение пришло из `ADMIN_CHAT_ID` и является reply.

Бот ищет в исходном сообщении строку `ID: <число>`. Если ID найден, отправляет текст reply-сообщения в чат клиента.

Acceptance criteria:

- Ответ администратора должен доставляться только при наличии ID клиента в replied-сообщении.
- Если ID не найден или replied-сообщение не содержит текст, бот ничего не отправляет.

## Диагностика волос

### Старт

Callback `test`:

- сбрасывает режим консультации;
- логирует действие `test`;
- создает прогресс с пустым списком ответов и сохраняет его в SQLite;
- отправляет стартовый вопрос из опубликованного `data/diagnostics_snapshot.json`.

### Ответ на вопрос

Callback имеет формат:

```text
test:<answer_id>:<next_question_id>
```

Бот:

- разбирает callback;
- восстанавливает прогресс из SQLite после перезапуска;
- проверяет, что ответ относится к ожидаемому вопросу;
- добавляет `answer_id` в список ответов;
- если `next_question_id == "advice"`, показывает результат;
- если `next_question_id` существует в вопросах, показывает следующий вопрос;
- иначе показывает результат.

### Результат

Бот не ищет точное совпадение полного пути ответов.

Новый алгоритм:

- берет список выбранных `answer_id`;
- читает смысл каждого ответа из `data/diagnostic_factors.json`;
- собирает labels, scores и tags;
- выбирает один основной модуль по явному запросу пользователя;
- добавляет актуальные стоп-флаги и не более двух контекстных дополнений;
- собирает HTML-рекомендацию в фиксированном формате без показа внутренних баллов;
- показывает клиентскую клавиатуру.

### Покрытие ответов

Текущее дерево диагностики содержит:

- 8 вопросов;
- 34 варианта ответа;
- 34 описания факторов в `diagnostic_factors.json`;
- 4 основных модуля, резервный модуль, 3 стоп-флага и 6 дополнений.

Все 3 696 валидных путей получают составную рекомендацию. Таблица контента проверяется полным прогоном перед публикацией.

## Общий текстовый fallback

Если пользователь отправляет текст, который не попал в режим консультации и не является командой, бот логирует пользователя и отвечает:

```text
Некорректная команда: <текст>
```

## Edge Cases

- Если JSON-файл отсутствует, `texts.py` возвращает пустой словарь или fallback-строку.
- Если ключ фото отсутствует, `file(key)` вернет строку `[no file: key]`, что может быть невалидным Telegram file ID.
- После перезапуска состояние консультации очищается, а диагностика восстанавливается из SQLite.
- Старая или повторно нажатая кнопка диагностики отклоняется и не учитывает ответ повторно.
- Если администратор отвечает не текстом, текущий код отправит `message.text`, которое может быть `None`.
