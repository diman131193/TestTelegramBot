# Спецификация тестирования

## Цели тестирования

- Проверить, что все пользовательские кнопки имеют обработчики или ожидаемое внешнее действие.
- Проверить, что бот не падает на отсутствующих или неполных JSON-данных.
- Проверить основные сценарии клиента, мастера, консультации и диагностики.
- Проверить сохранение пользователя в SQLite.

## Рекомендуемый уровень автоматизации

Для текущего размера проекта достаточно unit-тестов без реального Telegram API:

- тестировать `app.texts` на загрузку и fallback;
- тестировать `app.keyboards` на состав кнопок и callback-data;
- тестировать `app.db` на insert/update в временной SQLite-базе;
- тестировать дерево диагностики и факторную модель как чистые данные;
- мокать `Message`, `CallbackQuery` и `Bot` для handler-level тестов.

## Проверки данных

### `texts.json`

- Все callback-ключи, показываемые пользователю как экраны, имеют текст.
- Все кнопки имеют ключ `button_<key>`.
- HTML-разметка валидна для Telegram parse mode.

Текущий ожидаемый результат:

- `price` отсутствует и должен считаться найденной проблемой.

### `files.json`

- Для экранов с `answer_photo` есть file ID:
  - `start`;
  - `client`;
  - `services`;
  - `keratin`;
  - `botox`;
  - `nanoplastic`.
- `reviews` является списком строк.

### Опубликованный снимок и диагностические данные

- `start` существует в `questions`.
- Каждый `next`, кроме `advice`, указывает на существующий вопрос.
- Каждый option имеет `id`, `text`, `next`.
- Каждый option `id` имеет описание в `data/diagnostic_factors.json`.
- Все правила ссылаются на существующие карточки в `data/diagnostics_snapshot.json`.
- Для каждого пути выбирается один основной модуль, все стоп-флаги и не более двух дополнений.
- Каждый результат укладывается в лимит Telegram.

Текущее состояние:

- вопросов: 8;
- вариантов ответа: 34;
- описаний факторов: 34;
- терминальных путей: 3 696;
- все пути автоматически проверяются перед публикацией.

## Ручные smoke-тесты

### Старт

1. Отправить `/start`.
2. Ожидать фото, приветственный текст и две кнопки: `Мастер`, `Клиент`.
3. Проверить, что пользователь появился в `data/contacts.db`.

### Клиентское меню

1. Нажать `Клиент`.
2. Ожидать фото, текст с именем пользователя и меню клиента.
3. Проверить кнопки `Услуги`, `Прайс`, `Отзывы`, `Бесплатная диагностика`, `Запись`, `Консультация`.

### Услуги

1. Нажать `Услуги`.
2. Ожидать список услуг.
3. Нажать каждую услугу.
4. Ожидать фото, описание и меню возврата/следующих действий.

### Прайс

1. Нажать `Прайс`.
2. Текущий ожидаемый результат: бот не отвечает, потому что обработчик отсутствует.
3. После доработки: бот должен отправлять прайс или заглушку.

### Отзывы

1. Нажать `Отзывы` или отправить `/reviews`.
2. Ожидать media group с отзывами.
3. Ожидать текст отзывов и кнопку записи.

### Консультация

1. Нажать `Консультация`.
2. Отправить обычное текстовое сообщение.
3. Проверить, что администратор получил сообщение с ID клиента.
4. В админском чате ответить reply на сообщение бота.
5. Проверить, что клиент получил ответ.

### Диагностика

1. Нажать `Бесплатная диагностика`.
2. Ожидать первый вопрос.
3. Пройти путь:

```text
1.1 -> 2.1 -> 3.1 -> 4.1 -> 5.1 -> 8.1
```

4. Ожидать основной модуль про прикорневой объем и дополнение по способу сушки.
5. Пройти путь с другим основным запросом:

```text
1.1 -> 2.1 -> 3.2 -> 4.1 -> 5.1 -> 8.1
```

6. Ожидать основной модуль про свежесть корней и клиентскую клавиатуру.

## Автоматические тест-кейсы

### Клавиатуры

- `start_keyboard` содержит callback `master` и `client`.
- `client_keyboard` содержит callback `services`, `price`, `reviews`, `test`, `consulting` и URL-кнопку записи.
- `services_keyboard` содержит callback `keratin`, `botox`, `nanoplastic`.
- `services_menu_keyboard` содержит callback `price`, `reviews`, `consulting`, `client` и URL-кнопку записи.

### БД

- `init_db` создает таблицу `users`.
- `init_db` создает таблицу `diagnostic_sessions`.
- `log_user` вставляет нового пользователя.
- `log_user` обновляет существующего пользователя.
- `log_user` сохраняет старый `phone_number`, если новый не передан.
- `log_user` ничего не делает при `user is None`.

### Диагностика

- При callback `test` прогресс начинается с пустого списка ответов.
- При ответе callback `test:1.1:2` ответ добавляется в прогресс.
- При `next == advice` прогресс удаляется после результата.
- Повторный ответ на старую кнопку отклоняется.
- Прогресс восстанавливается из SQLite после очистки in-memory кеша.
- Для каждого валидного пути строится рекомендация через `build_recommendation`.
- Для path `1.1|2.1|3.2|4.1|5.4|8.1` выбирается `primary_oily_roots`.
- Для path `1.1|2.1|3.1|4.3|7.2|8.2` выбирается `primary_volume` и `addon_blow_dry`.
- Для path `1.4|2.3|3.4|4.2|6.2` выбираются `primary_dry_length`, `alert_chemical_damage` и `addon_long_hair`.

### Консультация

- Callback `consulting` добавляет chat ID в `ADMIN_CHATS`.
- Текстовое сообщение из `ADMIN_CHATS` отправляется администратору.
- `/start`, `client`, `services` и услуги удаляют chat ID из `ADMIN_CHATS`.
- Reply администратора с `ID: <chat_id>` отправляется клиенту.
- Reply администратора без ID игнорируется.

## Регрессионный чек-лист перед релизом

- Все JSON-файлы валидны.
- Нет callback-кнопок без обработчиков, кроме URL-кнопок.
- Все фото-ID актуальны в Telegram.
- `.env` содержит реальный `BOT_TOKEN`, а не placeholder.
- `.env` содержит `ADMIN_CHAT_ID`, и он соответствует реальному админскому чату.
- База `data/contacts.db` не коммитится с реальными данными.
- Служебная команда `/load` недоступна обычным пользователям или удалена.
